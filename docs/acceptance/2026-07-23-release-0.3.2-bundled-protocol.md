## 协议差异审查

- `ServerRequest` 保持 11 项，没有新增或删除服务端请求，失败关闭白名单无需变化。
- `DynamicToolCallParams`、`DynamicToolCallResponse` 的顶层字段和必填字段不变。
- `ThreadForkParams`、`ThreadStartParams`、`ThreadResumeParams`、
  `ThreadSettingsUpdateParams` 和 `TurnStartParams` 的顶层字段不变。
- `ClientRequest` 从 123 项增至 126 项，新增 `thread/searchOccurrences`、
  `app/read` 和 `app/installed`；Schema 描述均为线程搜索或应用元数据/已安装快照读取，
  不属于本地 Shell、文件、命令、进程或模糊文件搜索风险命名空间。

## 运行时接纳策略

- 官方扩展和内置 Codex 的版本值允许变化、未知或缺失；Controller 与 Shim 不比较
  `--version` 输出，也不把协议快照键当成运行时要求。
- `codexProtocolSnapshotVersion` 只选择开发期 Schema 快照并生成差异测试证据。
- Remote Executor ping 仍返回包版本和协议号供诊断，但 Controller 忽略其具体值，只
  要求当前远程读写、命令、探针和 stdio 能力集合完整。
- 只有官方内置二进制缺失、能力缺失、消息结构错误或真实操作失败才能阻断运行。

## 普通本地窗口实机证据

- 用户重载后，旧官方扩展可执行文件路径先产生一次 `ENOENT`。Controller 随即把
  `chatgpt.cliExecutable` 迁移到内容哈希对应的 `0.3.2` Shim，并自动触发第二次窗口重载；
  第二次启动完成 app-server `initialize`、当前 thread 恢复和 UI owner 绑定。该过程
  没有再出现 Codex、官方扩展、Executor 或协议版本不兼容提示。
- 活动进程确认使用 `0.3.2` Shim 和当前官方扩展目录内置 Codex；系统 Codex CLI 没有
  作为服务端启动。
- 显式 `codex-vscode` 与无参数普通 `codex` 都恢复同一 thread；后者从 CLI 发起真实
  `turn/start`，依次完成 `pwd`、CodeGraph `codegraph_status` 和最终
  `ACCEPTANCE_OK`。CLI 侧显示 Shell 与 MCP 工具过程及结果。
- Bridge 审计记录该外部客户端的 `initialize`、`thread/read`、`thread/resume`、
  `turn/start` 和正常断开；断开摘要确认向 VS Code 主客户端转发 22 条通知。官方
  Codex 日志记录相同 thread/turn 的 reasoning 与未读状态变化。
- 本次 `code --status` 仅发现一个普通本地 `NewNote` 窗口，没有活动 Remote SSH
  窗口。因此以上证据不能替代 G4、远端 G5 或 Remote Executor 生命周期验收。
