## Remote SSH 实机证据

- `code --status` 仅发现一个活动 `g1_1` Remote SSH transport；工作区根为
  `/home/unitree/mimiclite-sim2real`，远端为 Linux arm64 `unitree-g1-nx`。
- `0.3.3` 安装后，Controller 自动把托管 Shim 迁移到新的内容哈希目录并自动重载窗口；
  Bridge 随后进入 `ready`，审计记录相同远程主根和 `vscode-remote` 模式。
- 初始 VS Code thread 只有描述符但尚未物化，`thread/turns/list` 返回
  `not materialized yet`，`thread/resume` 返回 `no rollout found`。修复后的
  `codex-vscode` 自动改用顶层远程入口创建新的同步 thread，不再因空 rollout 退出。
- 外部 TUI 显示 `permissions: YOLO mode`，连续完成两轮。第二轮依次打印
  `remote_exec pwd`、`remote_read_file README.md` 和 CodeGraph
  `codegraph_status` 的开始、增量和结果，最终返回 `REMOTE_033_OK`。
- `pwd` 返回 `/home/unitree/mimiclite-sim2real`；README 读取以 `# sim2real` 开头；
  CodeGraph 返回 93 个文件、1758 个节点、3596 条边、5.19 MB 数据库和 WAL 模式。
- Bridge 审计记录 `remote_exec.approval` 为
  `automatic=true, permissionMode=full-access`，全程没有 Bridge 二次审批；外部连接
  断开时已向 VS Code 主客户端转发 141 条通知。
- 官方 Codex 日志记录同一 thread 的活动变化、两轮 reasoning 事件和
  `maybe_resume_success`；结构化读取确认该 thread 有 2 个已完成 turn，第二轮包含
  `mcpToolCall`。

## 遗留风险与结论

- 待补测：官方面板按钮发起的新任务、官方 UI 发起输入、取消、断线和附件/当前文件。
- 待补测：设置恢复、窗口关闭后的进程清理、Remote Executor 迁移及 OpenSSH 回退。
- 待补测：Windows x64 Controller 构建、包内容和实机。
- 已知限制：Linux 工作区没有 Windows SEA Shim；`npm run package:all` 因
  `dist/codex-bridge-shim.exe` 缺失而失败，不能从 Linux 结果推断双平台通过。
- 最终结论：限定范围通过。
- 支持声明：声明 Linux x64 上，外部 Codex CLI 通过 Bridge 复用活动 VS Code Remote
  SSH transport、新建同步 thread、进行多轮对话、显示远端工具过程并继承
  `full-access` 的链路通过；不声明 Windows、官方面板发起的新任务或完整生命周期通过。
