## 当前结论

`0.3.0` 的 MCP 控制面不是实时镜像。`0.3.1` 已完成普通本地窗口显式入口的同 thread
双向输入、事件和取消闭环，并补上普通 `codex` 下一次启动自动接入的安全启动链。此次
复核修复了一个此前被假服务器广播行为掩盖的缺口：每个外部 CLI 拥有独立上游连接，
官方 app-server 不保证把它的通知送到 VS Code 上游；现在由 Bridge 广播无 ID 通知并
去重，响应和服务端请求仍保持原连接路由。扩展安装状态变化会触发自动协调，Shim 迁移
后自动重载窗口。
当前已运行的旧 CLI 无法热切换；普通入口安装后本地实测已通过。权限、Remote SSH
回归和 Windows x64 闭环通过前仍只表示实施候选，不替代 `0.2.7` 已验收基线。
官方 UI 默认折叠“已处理”和“运行了多个命令”分组；这不代表工具事件未同步。展开
分组和单条工具记录后可以查看命令、MCP 名称、增量输出及完成状态。超过首次 exec
等待窗口的命令仍受官方 app-server 首段输出不生成 `commandExecution` 通知的限制；
Bridge 对其实际收到的后续事件保持逐条、实时、无改写转发。
